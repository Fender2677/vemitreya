# Keenetic — настройка для Vemitreya

Тестировалось на: **KeeneticOS 4.x** (Giga, Ultra, Hero, Hopper и т.д.).

## Особенность Keenetic

Keenetic **не имеет** полноценного firewall как у Mikrotik. Нет dst-nat по dst-address-list,
нет mark+route по spectrum-правил, нет policy routing в привычном виде.

Поэтому остаётся **только статические маршруты** для конкретных IP/подсетей — каждый IP проксируемого сервиса прописывается явно с `gateway=<Vemitreya IP>`.

Также схема **серверной части идентична**: `nft REDIRECT 80/443 → :7892` + `MASQUERADE`,
устанавливается автоматически в v2.206+ install.sh.

---

## Архитектура

```
Клиент LAN → Keenetic (статический маршрут на Telegram IP → gateway=сервер) →
   Vemitreya сервер (nft REDIRECT) → Mihomo → VPN канал
```

В отличие от Mikrotik (где один address-list + два mangle), здесь каждая подсеть Telegram/OpenAI/etc — отдельный статический маршрут. Это **рутинно но работает**.

| Переменная           | Пример              |
|----------------------|---------------------|
| `<VEMITREYA_IP>`     | `192.168.1.218`     |
| `<LAN_SUBNET>`       | `192.168.1.0/24`    |

## Автоматическая установка серверной части

**v2.206+** настраивает `nft REDIRECT + masquerade` автоматически.
Этот документ — только про Keenetic.

## Шаг 1. Резервация IP для Vemitreya сервера

В Веб-интерфейсе: **Мои сети и Wi-Fi** → **Список устройств** → выбрать сервер → **Зарегистрировать**.
Назначить статический IP (например `192.168.1.218`).

---

## Шаг 2. Статические маршруты

Способ через CLI (зайди в Keenetic по SSH или Telnet — `telnet 192.168.1.1`):

```
# Telegram
ip route 149.154.160.0/20 192.168.1.218 auto
ip route 91.108.4.0/22 192.168.1.218 auto
ip route 91.108.8.0/22 192.168.1.218 auto
ip route 91.108.12.0/22 192.168.1.218 auto
ip route 91.108.16.0/22 192.168.1.218 auto
ip route 91.108.20.0/22 192.168.1.218 auto
ip route 91.108.56.0/22 192.168.1.218 auto

# Meta
ip route 31.13.64.0/18 192.168.1.218 auto
ip route 157.240.0.0/17 192.168.1.218 auto
ip route 129.134.0.0/17 192.168.1.218 auto

# OpenAI
ip route 104.18.32.0/24 192.168.1.218 auto
ip route 104.18.33.0/24 192.168.1.218 auto
ip route 172.64.150.0/24 192.168.1.218 auto
ip route 172.64.154.0/24 192.168.1.218 auto
ip route 188.114.99.0/24 192.168.1.218 auto

# Anthropic
ip route 160.79.104.0/24 192.168.1.218 auto

# Сохранить
system configuration save
```

Или через Веб-интерфейс: **Интернет → Маршруты → "Добавить статический маршрут"** — но это утомительно для десятков подсетей.

## Шаг 3. Проверка

### На Keenetic
```
show ip route
# Должны быть строки с next-hop = 192.168.1.218
```

### На клиенте LAN
```bash
# Linux/Mac
traceroute -n 149.154.167.50
# Первый hop = Keenetic, второй = 192.168.1.218 (Vemitreya), третий+ = через VPN

# Windows
tracert 149.154.167.50
```

### На сервере Vemitreya
```bash
sudo journalctl -u mihomo -f | grep -v "xo.e0f"
# Видишь: [TCP] <client-ip>:YYYY --> some-host:443 match ...
```

## Особенности Keenetic

### Один интерфейс LAN
Все клиенты в одной подсети, бридж между Ethernet/Wi-Fi/Mesh. Это упрощает routing.

### DNS перехват
По умолчанию Keenetic перехватывает DNS-трафик к своему DNS-серверу — может ломать DOMAIN-SUFFIX правила Mihomo.

Отключить (если хочешь чтобы клиенты использовали DNS сервера которые сами выбрали):
```
no dns proxy
system configuration save
```

И на клиентах вручную поставить DNS = 1.1.1.1 или DNS-сервер Mihomo (:5353).

### Backup перед изменениями
Через Веб: **Управление → Резервные копии → Сохранить**.
Через CLI:
```
system configuration save
copy startup-config to startup-config-backup
```

## Альтернативные подходы

### 1. OPKG + sing-box/mihomo на самом Keenetic (entware)

Можно поставить sing-box/mihomo напрямую на Keenetic через entware — тогда **Vemitreya сервер не нужен** вовсе.

Это **другая** архитектура (без Vemitreya UI). См. документацию sing-box / mihomo и форумы Keenetic.

### 2. OpenVPN/WireGuard клиент Keenetic → Vemitreya

Завернуть **весь** трафик в один WG-туннель к Vemitreya. Но тогда:
- Правила Vemitreya не применятся (всё идёт одним каналом)
- Российские сайты тоже через VPN — медленнее, банят
- Не рекомендую если хочешь умное проксирование

### 3. DHCP option 121 для отдельных клиентов

Через DHCP-сервер Keenetic отдавать **только определённым** клиентам (по MAC) gateway = Vemitreya вместо Keenetic. Эти клиенты весь свой трафик отправят через сервер, и там Mihomo сам решит что куда.

Сложно настраивать и хрупко. Не рекомендую.

## Сравнение Mikrotik vs Keenetic для Vemitreya

| Аспект                          | Mikrotik             | Keenetic                |
|---------------------------------|----------------------|-------------------------|
| Address-list для роутинга       | ✅ Один список         | ❌ Каждый IP отдельным маршрутом |
| Mark-routing / policy routing    | ✅ Полноценно         | ❌ Нет                  |
| Простота настройки               | Средняя (CLI команды) | Простая (UI), но рутина |
| Гибкость                         | Высокая              | Низкая                  |
| Поддержка масштаба blocked_sites | Любой объём          | До ~50-100 маршрутов    |

**Для Vemitreya рекомендую Mikrotik**. Keenetic — если уже есть и менять не хочется,
и количество проксируемых подсетей ограничено десятками.

## Troubleshooting

| Симптом | Решение |
|---------|---------|
| Сайт не открывается | `traceroute` со клиента — второй hop должен быть 192.168.1.218 |
| | Если нет — статический маршрут не работает или подсеть IP не покрыта |
| Только некоторые сайты | IP сайта не в твоих статических маршрутах. Узнай через `dig` и добавь |
| Маршрут не активен | `show ip route` — может быть конфликт с автоматическим маршрутом |
| Server не отвечает на маршруте | Vemitreya сервер должен иметь IP_FORWARD=1 |
